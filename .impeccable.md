## Design Context

### Users
Personal finance tracker built by and for a single user today, but designed to be legible to anyone who might encounter it — not just the author. Users open it during daytime or evening sessions to review balances, log transactions, and check budgets. They're comfortable with basic financial concepts (accounts, budgets, tags) but the UI should not assume power-user fluency. PH-centric context: PHP currency (₱), Filipino time-of-day usage patterns.

### Brand Personality
**Calm · Trustworthy · Minimal**

The app handles money, which carries anxiety for most people. It should project stability and clarity — never urgency or visual noise. Every design decision should make the user feel *in control*, not overwhelmed. Personality is expressed through precision and restraint, not decoration or wit.

Emotional goal: the user opens the app and feels settled, not anxious. Numbers are presented clearly, context is always visible, and nothing demands unnecessary attention.

### Aesthetic Direction
- **Theme**: System-adaptive — both light and dark modes are first-class. Use DaisyUI's semantic color tokens throughout (`base-100`, `primary`, `error`, etc.) so neither mode is an afterthought.
  - Recommended starting themes: `silk` (light) + `dim` (dark). Both are subdued, professional, low-saturation — matches the Calm · Minimal direction without looking sterile.
- **Typography**: Clean, legible, monospace-friendly for numbers. Financial figures (`₱1,234.56`, account balances, budget percentages) should render in a tabular/mono-numeral style when available so columns align.
- **Color**: Restrained palette. Reserve `error`/red for genuinely negative states (overdrawn, over-budget). Avoid using color as decoration. Positive amounts can use `success` subtly; negative amounts use `error`. Neutral amounts use default text color.
- **Density**: Moderate — denser than a marketing page but not a spreadsheet. Tables use compact row height with comfortable horizontal padding. Cards use generous internal padding. Give data room to breathe without wasting screen real estate.
- **Shape**: Slightly rounded (`rounded-lg` / `rounded-xl`), consistent with DaisyUI defaults. No sharp-cornered brutalism, no pill-heavy softness.
- **Motion**: Minimal. Row flash on new transactions is appropriate. Avoid gratuitous transitions — the app is a tool, not a showcase.

### Design Principles

1. **Data first, chrome last.** The UI's job is to surface the user's numbers. Navigation, labels, and framing elements should recede visually. Hierarchy is achieved through size and weight, not decorative elements.

2. **Calm confidence.** Use color purposefully and sparingly. Reserve `error` red for states that genuinely need attention; never use it for routine negative amounts. Avoid ambient urgency — most states in a finance tracker are neutral, and they should feel that way.

3. **Legible to a stranger.** Every label, empty state, and button should be self-explanatory to someone who didn't build the app. No internal shorthand in the UI. Error messages explain what happened and what to do, not just what failed.

4. **Both modes are real.** Light and dark themes are both first-class surfaces. Never hardcode hex values — always use DaisyUI semantic tokens. Test both themes when shipping any UI change.

5. **Consistent rhythm.** Spacing, border-radius, and component sizing should be consistent across the app. Prefer DaisyUI component classes (`btn`, `badge`, `card`, `table`, `modal`) over custom utility stacks to stay in-system. Only diverge when the component genuinely doesn't exist.
